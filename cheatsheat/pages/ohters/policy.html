CORS

CORS, or Cross-Origin Resource Sharing, is a mechanism used in web development to control how resources on a web server can be accessed by web pages from a different domain (origin). It is implemented as part of web security standards to prevent unauthorized cross-origin HTTP requests, while still enabling legitimate interactions.

What Is an Origin?

An origin is defined by the combination of:

  • Protocol (e.g., http or https)
  • Domain (e.g., example.com)
  • Port (e.g., :80, :443)

Two URLs have the same origin if all three components are identical. For example:

  • https://example.com/page and https://example.com/data have the same origin.
  • http://example.com and https://example.com have different origins because of different protocols.

Why CORS Is Needed

Modern web browsers enforce a security policy called the Same-Origin Policy (SOP), which restricts how a web page loaded from one origin can interact with resources from another origin. SOP helps prevent malicious websites from accessing sensitive data on other sites. However, there are valid use cases where cross-origin requests are required, such as when a web application hosted on example.com needs to fetch resources from an API on api.example.com.

How CORS Works

CORS allows servers to specify which origins are permitted to access their resources. It involves setting specific HTTP headers in the server's response.

Key HTTP Headers in CORS

  1. Access-Control-Allow-Origin

    • Specifies which origin(s) are allowed to access the resource.
    • Example: Access-Control-Allow-Origin: https://example.com

  2. Access-Control-Allow-Methods

    • Lists HTTP methods (e.g., GET, POST) permitted for cross-origin requests.
    • Example: Access-Control-Allow-Methods: GET, POST

  3. Access-Control-Allow-Headers

    • Lists headers the client can include in the request.
    • Example: Access-Control-Allow-Headers: Content-Type, Authorization

  4. Access-Control-Allow-Credentials

    • Indicates whether credentials (e.g., cookies, authentication tokens) are allowed.
    • Example: Access-Control-Allow-Credentials: true

  5. Access-Control-Expose-Headers

    • Lists headers the client can access in the response.
    • Example: Access-Control-Expose-Headers: X-Custom-Header

  6. Access-Control-Max-Age

    • Specifies how long the results of a preflight request can be cached.
    • Example: Access-Control-Max-Age: 600 (10 minutes)

Types of CORS Requests

  1. Simple Requests

    • Made using methods like GET or POST without custom headers or complex content types.
    • CORS headers are validated in a straightforward manner.

  2. Preflight Requests

    • Required for requests that:
      • Use methods other than GET, POST, or HEAD.
      • Include custom headers.
    • Before sending the actual request, the browser sends an OPTIONS request to check if the server permits it.

CORS Example

  1. Client Request A browser attempts to fetch a resource from https://api.example.com/data while the web page is hosted on https://www.example.com.

  2. Server Response The server includes appropriate CORS headers, like:

    HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://www.example.com
Access-Control-Allow-Methods: GET, POST

  3. Browser Enforcement If the headers match the browser's expectations, the request proceeds. Otherwise, it's blocked.

Summary

CORS is an essential web standard that balances security and flexibility. It allows legitimate cross-origin communication while ensuring unauthorized or malicious access is restricted. Proper server-side configuration of CORS headers is critical to avoid security risks while enabling functionality.
