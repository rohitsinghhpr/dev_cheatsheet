1xx - Informational
The 1xx series represents informational responses. These are provisional responses to indicate that the request has been received and is being processed.
  • 100 Continue: The client should continue with the request.
  • 101 Switching Protocols: The server is switching protocols as requested by the client.
  • 102 Processing: The server has received the request and is processing it, but no response is available yet (WebDAV).
  • 103 Early Hints: Used to send headers before the final response (primarily for performance reasons).
2xx - Successful
The 2xx series indicates successful responses. These codes confirm that the client's request was successfully received, understood, and accepted.
  • 200 OK: The request was successful and the server responded with the requested resource.
  • 201 Created: The request was successful, and the server created a new resource.
  • 202 Accepted: The request has been accepted for processing, but the processing is not complete.
  • 203 Non-Authoritative Information: The request was successful, but the returned information is from a cache or a third-party source.
  • 204 No Content: The request was successful, but there is no content to return.
  • 205 Reset Content: The server successfully processed the request, but the client should reset the document view.
  • 206 Partial Content: The server is delivering only a portion of the resource, typically in response to a range request.
3xx - Redirection
The 3xx series indicates that further action is required to complete the request. The client must follow the redirection to the new resource.
  • 300 Multiple Choices: The request has more than one possible response. The client should choose one of them.
  • 301 Moved Permanently: The requested resource has been permanently moved to a new location, specified by the Location header.
  • 302 Found: The requested resource is temporarily located at a different URI, but future requests should still use the original URI.
  • 303 See Other: The response can be found under a different URI, and the client should use a GET method to retrieve it.
  • 304 Not Modified: The resource has not been modified since the last request, and the client can use its cached copy.
  • 305 Use Proxy: The requested resource must be accessed through the specified proxy.
  • 306 (Unused): This code was previously used, but it is no longer in use.
  • 307 Temporary Redirect: The requested resource is temporarily available at a different URI, and the client should use the original URI for future requests.
  • 308 Permanent Redirect: The requested resource has been permanently moved to a new location, and future requests should use the new URI.
4xx - Client Errors
The 4xx series represents client errors. These codes indicate that the request was malformed or cannot be processed due to issues with the client.
  • 400 Bad Request: The request was malformed or invalid, and the server cannot process it.
  • 401 Unauthorized: The request requires user authentication, and the client has not provided valid credentials.
  • 402 Payment Required: Reserved for future use. It was originally intended for payment systems.
  • 403 Forbidden: The server understood the request, but it refuses to authorize it.
  • 404 Not Found: The requested resource could not be found on the server.
  • 405 Method Not Allowed: The HTTP method used is not supported for the requested resource.
  • 406 Not Acceptable: The requested resource is not available in a format that is acceptable to the client.
  • 407 Proxy Authentication Required: The client must authenticate itself with the proxy server before the request can be fulfilled.
  • 408 Request Timeout: The server timed out waiting for the client to send a request.
  • 409 Conflict: The request could not be completed due to a conflict with the current state of the resource.
  • 410 Gone: The requested resource is no longer available and has been permanently removed from the server.
  • 411 Length Required: The server requires the `Content-Length` header, but it was not provided by the client.
  • 412 Precondition Failed: One of the conditions in the request header field was evaluated as false.
  • 413 Payload Too Large: The request entity is larger than the server is willing or able to process.
  • 414 URI Too Long: The URI provided in the request is too long for the server to process.
  • 415 Unsupported Media Type: The server cannot process the media type of the request payload.
  • 416 Range Not Satisfiable: The client has requested a range that is not available for the resource.
  • 417 Expectation Failed: The server cannot meet the expectation specified in the `Expect` header.
  • 418 I'm a teapot: A humorous code from the "Hyper Text Coffee Pot Control Protocol," indicating the teapot is not capable of making coffee.
  • 421 Misdirected Request: The request was directed at a server that is not able to produce a response.
  • 422 Unprocessable Entity: The request is well-formed but cannot be followed due to semantic errors (WebDAV).
  • 423 Locked: The resource that is being accessed is locked (WebDAV).
  • 424 Failed Dependency: The request failed due to a failure of a previous request (WebDAV).
  • 425 Too Early: The server is unwilling to risk processing a request that might be replayed (HTTP/2).
  • 426 Upgrade Required: The client must upgrade to a different protocol to complete the request.
  • 427 Unassigned: Reserved for future use.
  • 428 Precondition Required: The origin server requires the request to be conditional.
  • 429 Too Many Requests: The user has sent too many requests in a given amount of time.
  • 431 Request Header Fields Too Large: The server is refusing to process the request because the header fields are too large.
  • 451 Unavailable For Legal Reasons: The resource is unavailable due to legal reasons (e.g., censorship).
5xx - Server Errors
The 5xx series indicates server errors. These codes mean that the server failed to fulfill a valid request.
  • 500 Internal Server Error: The server encountered an unexpected condition that prevented it from fulfilling the request.
  • 501 Not Implemented: The server does not support the functionality required to fulfill the request.
  • 502 Bad Gateway: The server, while acting as a gateway or proxy, received an invalid response from the upstream server.
  • 503 Service Unavailable: The server is currently unavailable (due to overload or maintenance).
  • 504 Gateway Timeout: The server, while acting as a gateway or proxy, did not receive a timely response from the upstream server.
  • 505 HTTP Version Not Supported: The server does not support the HTTP protocol version that was used in the request.
  • 506 Variant Also Negotiates: The server has an internal configuration error and cannot perform content negotiation (RFC 2295).
  • 507 Insufficient Storage: The server cannot store the representation needed to complete the request (WebDAV).
  • 508 Loop Detected: The server detected an infinite loop while processing a request (WebDAV).
  • 510 Not Extended: The server requires further extensions to fulfill the request (RFC 2774).
General HTTP Headers
General headers are used in both request and response messages and provide essential information about the communication.
  • Cache-Control: Directives for caching mechanisms in both requests and responses.
  • Connection: Control options for the current connection (e.g., `keep-alive`, `close`).
  • Date: The date and time at which the message was sent (in HTTP response headers).
  • Content-Encoding: Specifies any encodings that have been applied to the response body (e.g., `gzip`, `deflate`).
  • Content-Length: The length of the response body in octets (8-bit bytes).
  • Content-Type: The MIME type of the resource (e.g., `text/html`, `application/json`).
  • Host: The domain name of the server (required in HTTP/1.1).
  • Transfer-Encoding: Specifies the form of encoding used to safely transfer the response body to the user.
Request Headers
Request headers are used by the client to provide information about the request or about the client itself.
  • Accept: Specifies the media types that are acceptable for the response (e.g., `application/json`, `text/html`).
  • Accept-Encoding: Indicates the content encodings (such as `gzip`, `deflate`) that the client can understand.
  • Authorization: Contains the credentials for HTTP authentication (e.g., `Basic`, `Bearer`).
  • Cookie: Sends stored cookies from the client to the server.
  • Host: Specifies the domain name of the server and the TCP port number.
  • If-Modified-Since: Allows the server to send back a 304 Not Modified if the resource has not changed since the given date.
  • Referer: Provides the URL of the resource that triggered the request.
  • User-Agent: Identifies the client software making the request (e.g., browser type, operating system).
  • Content-Type: The type of data sent in the body of the request (e.g., `application/json`, `application/x-www-form-urlencoded`).
  • Content-Length: The size of the request body in octets (8-bit bytes).
Response Headers
Response headers provide information about the server's response and control how the response is processed.
  • Content-Type: The MIME type of the response (e.g., `text/html`, `application/json`).
  • Content-Length: The size of the response body in octets (8-bit bytes).
  • Set-Cookie: Sends cookies from the server to the client to store for future requests.
  • Location: Used in redirections to specify the URI of the new resource (e.g., `301 Moved Permanently`).
  • Server: Contains information about the server software handling the request (e.g., `Apache`, `nginx`).
  • Cache-Control: Directives for caching mechanisms in the response.
  • Expires: Specifies the date and time after which the response is considered stale.
  • Access-Control-Allow-Origin: Specifies which domains are permitted to access the resource via Cross-Origin Resource Sharing (CORS).
  • Content-Encoding: Indicates the encoding transformations applied to the response body (e.g., `gzip`, `deflate`).
  • ETag: A unique identifier for a specific version of a resource, used for caching purposes.
Authentication Headers
Authentication headers are used for security and controlling access to resources.
  • Authorization: Used by the client to provide credentials for HTTP authentication, often in the format `Basic` or `Bearer` tokens.
  • WWW-Authenticate: Used in the response to challenge the client for authentication credentials (e.g., `Basic realm="Example"`).
Cache-Control Headers
Cache-Control headers are used to define caching policies in both requests and responses.
  • Cache-Control: Specifies directives for caching in both requests and responses, such as `no-cache`, `max-age`, or `public`.
  • Pragma: A legacy HTTP header used for caching purposes (e.g., `Pragma: no-cache`).
  • Expires: Specifies the date and time after which the response is considered stale (used for caching).
  • If-None-Match: Used to determine if the content has changed by comparing the client's stored ETag value.
  • If-Modified-Since: Tells the server to send back a 304 response if the content has not changed since the specified date.
Content Negotiation Headers
Content negotiation headers are used by clients and servers to negotiate the format of the response.
  • Accept: Specifies the media types that the client is willing to accept in the response (e.g., `application/json`, `text/html`).
  • Accept-Encoding: Specifies the content encodings the client is able to accept (e.g., `gzip`, `deflate`).
  • Accept-Language: Specifies the language(s) the client prefers for the response.
  • Content-Type: Specifies the media type of the body of the request (e.g., `application/json`, `text/xml`).
What are the HTTP methods used in API requests?
HTTP methods define the actions that can be performed on resources in an API. The primary methods are:
  • GET: Retrieves data from the server. It is a safe and idempotent method, meaning it does not alter data.
  • POST: Sends data to the server to create or update a resource. It is not idempotent, as calling it multiple times can result in different outcomes.
  • PUT: Replaces a resource with new data. It is idempotent, meaning that calling it multiple times with the same data will have the same effect.
  • PATCH: Partially updates a resource. Like PUT, it modifies an existing resource but only updates the specified fields.
  • DELETE: Deletes a resource from the server.
  • OPTIONS: Returns the HTTP methods that are allowed for a specific resource.
  • HEAD: Similar to GET, but it only retrieves the headers, not the body of the response. It's often used to check for the presence of a resource.
What is the GET method in HTTP?
The GET method is used to request data from a specified resource. It is the most common HTTP method used in APIs to retrieve information.
  • Characteristics: It is a safe and idempotent method, meaning it does not modify the resource and can be safely repeated without causing side effects.
  • Usage: Commonly used to retrieve data such as user profiles, product details, and search results.
  • Example: GET /api/v1/users/123 retrieves information about user with ID 123.
Example: 

                    GET /api/v1/products HTTP/1.1
                    Host: example.com
What is the POST method in HTTP?
The POST method is used to send data to the server to create or update a resource. Unlike GET, POST is not idempotent, meaning that multiple identical requests can result in different outcomes.
  • Characteristics: It is typically used to create resources or submit forms.
  • Usage: Commonly used for actions like user login, creating a new product, or submitting an order.
  • Example: POST /api/v1/users creates a new user.
Example: 

                    POST /api/v1/orders HTTP/1.1
                    Host: example.com
                    Content-Type: application/json
            
                    {
                        "product_id": 1,
                        "quantity": 2
                    }
What is the PUT method in HTTP?
The PUT method is used to update a resource or create a new resource if it doesn't exist. It replaces the entire resource with the new data sent in the request body.
  • Characteristics: PUT is idempotent, meaning that making the same request multiple times will always produce the same result.
  • Usage: It is used when the client wants to completely replace a resource with new data.
  • Example: PUT /api/v1/users/123 replaces the data of user 123 with new data provided in the request body.
Example: 

                    PUT /api/v1/products/1 HTTP/1.1
                    Host: example.com
                    Content-Type: application/json
            
                    {
                        "name": "Updated Product",
                        "price": 20.99
                    }
What is the PATCH method in HTTP?
The PATCH method is used to partially update a resource. Unlike PUT, which replaces the entire resource, PATCH only updates the fields provided in the request.
  • Characteristics: PATCH is not idempotent by default, meaning repeated requests with the same data may have different results if the data is not handled correctly.
  • Usage: It is commonly used when updating a single field of a resource, such as changing a user's email address or updating the status of an order.
  • Example: PATCH /api/v1/users/123 updates the user with ID 123 with new data in the request body.
Example: 

                    PATCH /api/v1/users/123 HTTP/1.1
                    Host: example.com
                    Content-Type: application/json
            
                    {
                        "email": "newemail@example.com"
                    }
What is the DELETE method in HTTP?
The DELETE method is used to delete a resource on the server. It removes the specified resource, and if successful, it generally returns a status code indicating that the resource was deleted.
  • Characteristics: It is idempotent, meaning that calling DELETE multiple times on the same resource will have the same effect (the resource will remain deleted).
  • Usage: Commonly used to delete a user, product, or any other resource from a system.
  • Example: DELETE /api/v1/users/123 deletes the user with ID 123.
Example: 

                    DELETE /api/v1/products/1 HTTP/1.1
                    Host: example.com
What is the OPTIONS method in HTTP?
The OPTIONS method is used to describe the communication options for the target resource. It allows the client to determine which HTTP methods are supported by a specific resource.
  • Characteristics: It does not require a request body and typically returns an HTTP response header listing supported methods (e.g., GET, POST, PUT, DELETE).
  • Usage: Often used for CORS (Cross-Origin Resource Sharing) requests and to discover allowed methods for a specific endpoint.
  • Example: OPTIONS /api/v1/products returns the allowed HTTP methods for the products resource.
Example: 

                    OPTIONS /api/v1/products HTTP/1.1
                    Host: example.com
What is the HEAD method in HTTP?
The HEAD method is used to retrieve the headers of a resource, but it does not return the body. It is often used to check for the existence of a resource or get metadata (such as the length of the content or last-modified date).
  • Characteristics: It is similar to GET, but it only retrieves the headers of the response, not the resource data.
  • Usage: Commonly used to check if a resource exists without downloading it or to obtain metadata about a resource.
  • Example: HEAD /api/v1/products/123 retrieves only the headers of the product with ID 123.
Example: 

                    HEAD /api/v1/products/123 HTTP/1.1
                    Host: example.com
What are idempotent and safe HTTP methods?
  • Idempotent Methods: An HTTP method is idempotent if making multiple identical requests has the same effect as making a single request. For example, PUT and DELETE methods are idempotent because repeating the same request will not have additional effects.
  • Safe Methods: A safe method is one that does not alter the resource. GET, HEAD, OPTIONS, and TRACE are considered safe because they only retrieve data without changing it.
